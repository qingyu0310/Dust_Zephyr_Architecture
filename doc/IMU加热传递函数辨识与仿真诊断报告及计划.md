## 1. 结论摘要

### 1.1 当前最核心的问题

当前的 `imu_ctrl_log.py` 注释写的是“开环拟合”，但默认采集路径并不是开环实验。`modules/imu/imu.cpp` 在普通工作模式下使用 `Pid::Calc()` 闭环计算 duty，`imu_ctrl_log.py` 只是读取闭环已经产生的 `temperature` 和 `duty`，再用一个植物模型复现温度。因此它不是严格意义上的开环辨识，而是“已知控制器作用下的闭环输入输出反演”。

闭环数据并非绝对不能用于辨识，但必须知道：控制器会主动压低误差，导致输入 duty 与温度之间的有效激励不足，且参数之间可能互相补偿。当前脚本没有检查输入激励、参数可辨识性、训练集与验证集分离，也没有输出参数置信区间，所以拟合 MSE 小不能证明 `K/tau/delay` 是真实的植物参数。

### 1.2 当前最可能直接污染辨识结果的问题

1. 固件辨识阶段的“稳定”判定不是热稳态判定。`UpdateIdSweep()` 中的参考温度会随着温度变化被更新，`kIdStableSamples = 500` 只表示温度连续落在动态参考值附近，不表示温度斜率接近零。
2. `imu_temp_identify.py` 的分段曲线拟合把每个阶段当成从固定输入阶跃开始的独立响应，但固件在阶段之间会回到 `kCoolingDuty = 0.001`，且不会回到同一温度。阶段初始温度、环境散热状态和历史热量都在变化。
3. `imu_ctrl_log.py` 用单个 `median(diff(time))` 代表整段数据的采样周期，并用离散 `delay_n` 量化延迟。串口日志时间戳、日志丢帧和实际控制周期抖动都会被压进模型参数，尤其会把采样/打印/调度误差伪装成 delay。
4. `tune_imu_pi.py` 声称是参数 sweep，但 `main()` 只仿真固定的 `PI_KP`、`PI_KI`，既没有调用 `build_grid()`，也没有遍历搜索范围，因此当前并没有完成 PI 参数扫描。
5. 仿真里的积分周期写死为 `0.001`，没有使用函数参数 `dt`；当命令行传入其他 `--dt` 时，植物离散化变了，控制器积分仍然按 1 ms 积分，仿真语义不一致。

### 1.3 推荐的总方向

不要先继续调 `K/tau/delay`，而应按以下顺序推进：

1. 先修正采集语义：记录真实时间、实际 duty、阶段事件、温度和采样周期，并确认阶段确实是可辨识的输入激励。
2. 再做专门的开环实验：使用多个 duty 阶跃或 PRBS/多级阶跃，禁止 PI 参与同一段辨识。
3. 使用真实采样时间的离散状态空间模型拟合，先从一阶无延迟、再到一阶加延迟，必要时才升级二阶热模型。
4. 用完全独立的新实验验证，而不是用拟合时使用的同一段数据验证。
5. 最后才把已验证植物接入 PI 仿真，并逐项对齐固件 PID、PWM 限幅、积分状态和实际闭环延迟。

## 2. 当前代码链路

### 2.1 普通闭环路径

当前普通模式的实际链路是：

```text
IMU Read(sample.temperature)
    -> heater::Heater::Update()
    -> alg::pid::Pid::Calc(target=40, now=temp)
    -> duty 限幅 [0.001, 0.90]
    -> PWM SetDuty()
    -> 热对象
    -> 下一次温度采样
```

具体位置：

- `modules/imu/imu.cpp:340-355`：`Heater::Update()` 在 `kDisabled` 状态调用 `pid_.Calc()`，随后把 duty 限制到 `[kMinDuty, kMaxDuty]`。
- `algorithm/controller/pid/pid.cpp:78-155`：PID 完成积分、输出限幅和状态保存。
- `modules/imu/imu.cpp:168-171`：普通模式日志输出 `temp` 和 `duty`。
- `modules/imu/imu.cpp:155-166`：IMU 线程以驱动建议周期轮询，并在姿态处理前测量实际经过时间；但 heater PID 自己仍使用配置中的固定 `dt=0.001`。

### 2.2 固件自动辨识路径

自动辨识模式的路径是：

```text
EnableIdSweep()
    -> waiting_for_stable: duty = 0.001
    -> heating: duty = 0.35, 0.40, 0.45 ...
    -> 温度“稳定”或达到 70 C
    -> stage_end
    -> duty = 0.001 冷却
    -> 下一阶段
```

具体位置：

- `modules/imu/imu.hpp:98-105`：阶段参数为起始温度 36 C、duty 0.35 到 0.45、步长 0.05、稳定容差 0.10 C、安全温度 70 C。
- `modules/imu/imu.cpp:393-471`：辨识状态机。
- `modules/imu/imu.cpp:357-365`：加热阶段日志，日志内容包含温度、duty、`sample.dt` 和稳定计数。
- `modules/imu/imu.cpp:434-463`：阶段开始、阶段结束、切换下一阶段。

### 2.3 三个 Python 脚本各自实际做的事

| 脚本 | 当前行为 | 它真正能回答的问题 | 当前不能证明的问题 |
|---|---|---|---|
| `imu_temp_identify.py` | 按 duty 变化切段，每段拟合 `T=T0+dT(1-exp(-(t-L)/tau))`，也拟合共享 `tau/L` | 某一段数据是否近似一阶延迟曲线 | 共享参数是否跨实验有效，duty 到温升是否线性，阶段是否真的达到稳态 |
| `imu_ctrl_log.py` | 用测得 duty 驱动一阶模型，优化 `K/tau/delay` 使温度 MSE 最小 | 该模型能否回放当前这段 duty/温度序列 | 参数是否可辨识，是否是开环植物参数，是否能预测新的闭环响应 |
| `tune_imu_pi.py` | 用固定模型仿真一个固定 PI，计算超调、首次到达、稳定时间等指标 | 该组固定参数在该模型下表现如何 | 参数搜索是否完成，模型与固件是否一致，真机是否会得到同样响应 |

## 3. 模型与数据语义

### 3.1 当前假设的一阶带纯延迟模型

三个脚本围绕的模型可写为：

```text
G(s) = K / (tau*s + 1) * exp(-delay*s)
```

这里的输入是 duty，输出通常被理解为温度相对环境温度的变化。若 `K` 的单位是摄氏度/duty，则在恒定 duty `u` 下的稳态温度应近似为：

```text
T_ss = T_ambient + K*u
```

这个模型只在以下条件同时近似成立时才合理：

- 加热器功率与 duty 近似线性；
- 温度传感器响应、加热器响应和热扩散可以用一个主导时间常数表示；
- 散热与温度的关系在工作区间内可以线性化；
- 环境温度和机械安装条件基本不变；
- 延迟主要是固定延迟，而不是温度相关的传感器/计算/日志延迟；
- 输入没有 PWM 饱和、输出死区和明显非线性。

IMU 加热器通常存在对流散热、安装接触热阻、IMU 芯片内部热容、板级热扩散和温度传感器位置差异。因此一阶模型可以作为控制设计的低阶近似，但不能未经交叉验证就当成物理真值。

### 3.2 开环、闭环和回放的区别

三种数据用途必须分开：

1. **开环辨识**：人为给定 duty，记录温度响应，用于估计植物 `G`。此时控制器不能同时根据温度修改 duty。
2. **闭环验证**：给定目标温度和控制器，记录真机闭环响应，用于验证 `G + C` 的闭环预测。
3. **输入回放**：把真机记录的 duty 原样送入候选植物模型，比较温度轨迹。它可检验模型预测能力，但闭环记录本身不能自动变成开环辨识数据。

`imu_ctrl_log.py` 当前做的是第 3 类，却在注释中称为第 1 类，这是报告中必须纠正的概念。

## 4. 确定性问题

本节的问题可以直接从当前代码确认，不依赖某一份尚未提供的日志。

### 4.1 固件稳定判定会提前结束阶段

位置：`modules/imu/imu.cpp:395-414`。

当前逻辑是：

```cpp
if (IsStable(current_tempe, stable_reference_temp_, tolerance)) {
    stable_count_++;
} else {
    stable_count_ = 0;
    stable_reference_temp_ = current_tempe;
}
```

问题在于 `stable_reference_temp_` 在任何一次超出窗口后都会被设置成当前温度。只要温度曲线是平滑上升的、相邻样本变化小于 0.10 C，就可能连续计数 500 次，即使温度仍然有明显正斜率。

这不是“温度稳定在目标值附近”，而是“相邻采样的温度变化没有超过一个很宽的窗口”。对于 1 ms 采样、温度输出只有 0.01 C 或 0.02 C 分辨率的情况，这个条件尤其容易误判。

**影响：**

- 每个阶段的结束时刻取决于传感器分辨率和日志/采样行为，而不是热对象真实稳态；
- `delta_temp` 会被低估或变成阶段截断时的暂态温升；
- `tau` 和 `delay` 会被迫补偿错误的阶段边界；
- 不同阶段的结果不可直接比较。

**合理判据：**至少同时要求：

```text
窗口内温度均值接近目标或当前阶段的稳态估计
窗口线性拟合斜率 |dT/dt| < slope_limit
窗口温度标准差 < noise_limit
持续时间 >= N 秒
```

### 4.2 阶段起始温度和初始条件并不统一

位置：`modules/imu/imu.hpp:99`、`modules/imu/imu.cpp:419-435`。

注释将 `kIdStartTemp=36 C` 描述为“辨识阶段允许的最高起始温度”，代码要求温度不高于约 36.1 C，然后利用固定窗口稳定判定进入加热。它不再强制所有阶段精确从同一个温度开始，但 `stage_start,temp` 会记录每个阶段真正的初始温度；板温、环境温度和冷却时间仍需实验记录。

阶段之间还会执行 `duty=0.001` 冷却，但下一阶段并不保证回到同一个热状态。热惯性意味着：即使表面温度相同，芯片内部和 PCB 内部的温度分布也可能不同。

**影响：**把不同阶段的绝对温度曲线直接共享 `tau/L`，会把历史热状态差异误认为模型参数差异。

### 4.3 `imu_temp_identify.py` 的切段假设和固件实验不匹配

位置：`scripts/imu/imu_temp_identify.py:451-496`。

`split_segments()` 按 duty 发生变化切段，但它不会把 duty 从 0.35 切到 0.001 的冷却过程单独建模，也不会利用 `stage_start` 和 `stage_end` 事件恢复准确的阶段边界。对于 CSV，阶段边界只取决于相邻 duty 是否变化。

而固件实际的阶段变化是：

```text
阶段结束 -> 先把 duty 设为 0.001 -> 等待 -> 下一阶段 duty
```

因此脚本得到的每个段可能是“某个 duty 阶段的一部分”，也可能包含阶段切换附近的采样点。它没有建模每个阶段的真实输入初值和热状态。

### 4.4 `imu_ctrl_log.py` 的环境温度被硬编码为 35 C

位置：`scripts/imu/imu_ctrl_log.py:338-341`。

文件顶部定义了 `AMBIENT_TEMPERATURE`，注释写着 `None = 自动取第一条温度值`，但 `main()` 没有使用它，而是直接执行：

```python
ambient = RISE_TEMP
```

且 `RISE_TEMP = 35.0`。如果第一条记录不是环境稳态、设备刚经历了冷却、或者真实环境温度不是 35 C，拟合会通过改变 `K/tau/delay` 补偿错误的基线。

这还是一个更深层的建模问题：仅凭一段闭环升温日志，环境温度和初始温度通常不能同时可靠估计。必须在实验中单独记录环境/板级参考温度，或者把基线作为已知量并在实验前等待热状态明确。

### 4.5 `imu_ctrl_log.py` 的 delay 只能被量化到一个采样点

位置：`scripts/imu/imu_ctrl_log.py:239-248`。

`delay_n = round(delay/dt)`，且至少为 1；当 `dt` 为 1 ms 时，delay 的搜索结果只能以约 1 ms 为步进。更重要的是，代码在每次更新时先读取 buffer 当前值，再写入当前 duty，因此具体的延迟定义还受到缓冲区长度和样本索引影响。

如果输入 duty 是闭环控制器实时产生的，温度日志是异步打印的，那么这里估计出的 `delay` 混合了：

- 加热器到 IMU 温度的真实热延迟；
- 传感器内部滤波延迟；
- 线程调度延迟；
- 日志时间戳粒度和排队延迟；
- duty 日志代表“更新前”还是“更新后”的时序偏差。

因此 `delay=1.793 s` 不能直接解释为纯热延迟。

### 4.6 时间戳处理没有做数据质量检查

位置：`scripts/imu/imu_ctrl_log.py:54-72`、`scripts/imu/imu_temp_identify.py:184-205`。

当前代码没有拒绝或报告以下情况：

- 时间戳不递增；
- 时间戳重复；
- 日志丢行造成的大间隔；
- 回绕或跨小时；
- `dt=0` 或异常大 `dt`；
- 样本时间与温度/duty 的实际采样时间不一致。

`imu_temp_identify.py` 甚至用 `max(timestamp_s-last_timestamp_s, 0.0)` 静默吞掉倒退时间。这样会让坏数据继续进入拟合，而不是让实验者知道数据不可信。

### 4.7 `tune_imu_pi.py` 没有执行参数扫描

位置：`scripts/imu/tune_imu_pi.py:230-233` 和 `:236-335`。

`build_grid()` 定义了几何网格，但没有被调用。`main()` 直接使用 `PI_KP` 和 `PI_KI` 做一次仿真，也没有生成候选参数排名。因此命令行的 `--kp-min`、`--kp-max`、`--ki-min`、`--ki-max`、`--grid-size` 当前都是无效配置。

这会导致一个常见误解：输出了一组 PI 指标，不代表它是搜索得到的最优组，也不代表当前模型下的控制器已经调优。

同一处还有一个指标语义问题：`first_reach_time` 虽然被计算并打印，但没有参与 `score`。因此如果目标是“尽快达到目标温度”，当前评分并不会直接优化首次到达时间；它主要优化的是稳定时间、稳态误差、超调和饱和比例。

### 4.8 `tune_imu_pi.py` 的控制积分没有使用 `dt`

位置：`scripts/imu/tune_imu_pi.py:139`。

代码写的是：

```python
integral_error += 0.001 * current_error
```

正确的语义应当使用传入的 `dt`。当前命令行允许 `--dt`，但只改变温度更新和 delay 采样，不改变积分累加。因此只要 `--dt != 0.001`，仿真就不再对应固件或一个一致的离散 PI。

### 4.9 固件 PID 的 `dt` 与真实控制循环也不是同一个量

位置：`modules/imu/imu.hpp:106-114`、`modules/imu/imu.cpp:155-166`、`algorithm/controller/pid/pid.cpp:78-155`。

IMU 线程会测量真实 sample dt 并写入 `sample_.dt`，但 heater PID 的配置仍固定为 `dt=0.001`，`Heater::Update()` 也没有把真实 dt 传给 PID。对于 D 项为 0、采样周期非常稳定的当前配置，这个差异可能不大；但它仍然意味着：

- 仿真用的是固定 1 ms 控制器；
- 姿态链使用的是测得 dt；
- 固件 heater 闭环实际受到调度抖动影响，却没有记录真实控制更新间隔。

辨识报告必须记录实际样本间隔分布，而不能只引用名义 1 ms。

### 4.10 `fit_plant()` 没有约束增益的物理范围

位置：`scripts/imu/imu_ctrl_log.py:263-268`。

优化器只拒绝 `tau <= 0` 和 `delay < 0`，没有拒绝 `K <= 0`，也没有给 `K`、`tau`、`delay` 设置上界。对于加热器，正常工作区间内应首先假设 duty 增大不会让温度响应方向反转，因此 `K` 应为正；延迟和时间常数也应有基于实验窗口的合理上界。

不加边界时，优化器可能用异常参数解释基线错误、阶段截断或坏时间戳。即使最终得到正的 `K`，也不能说明搜索过程受到了物理约束。建议使用带 bounds 的优化方法，并把“参数越界”作为实验/模型失败信号输出，而不是静默接受。

## 5. 建模与实验设计风险

这些不一定是代码 bug，但会让“拟合很好”失去工程意义。

### 5.1 闭环输入激励不足导致参数不可辨识

闭环中 duty 由误差决定：温度越接近目标，duty 越趋于维持温度所需的值。若整段日志只覆盖 0.1 到 0.2 的 duty 小范围，很多组 `K/tau/delay` 都可能给出相近温度轨迹。

辨识前应检查：

- duty 的范围和标准差；
- duty 的阶跃数量和幅度；
- duty 与温度的互相关；
- 输入是否长期饱和在最小或最大 duty；
- 输入是否被控制器和限幅器强烈塑形。

如果输入没有足够激励，正确结论应是“本次数据不能可靠辨识”，而不是继续优化参数。

### 5.2 一阶模型可能漏掉热对象的第二个主导时间常数

常见热链路可能至少包含：

```text
加热器 -> PCB/封装 -> IMU 芯片 -> 传感器温度读数
```

如果响应先快后慢，单个 `tau` 会在早期和后期之间折中。此时同一组参数可能在均方误差上不错，但在闭环控制关注的延迟、首次到达和小误差段上很差。

应比较以下候选模型，而不是预设一阶一定正确：

```text
M1: 一阶无延迟
M2: 一阶 + 纯延迟
M3: 二阶串联 + 纯延迟
M4: 一阶 + 温度相关散热系数
```

模型升级必须由独立验证误差和残差结构驱动，不能只看训练集 MSE。

### 5.3 `K` 不一定对 duty 线性

PWM duty 到实际平均加热功率之间可能有：

- MOSFET/PWM 输出非线性；
- 低 duty 死区；
- 电源电压变化；
- 加热器电阻随温度变化；
- 热损耗随温度增加而变大。

因此多 duty 阶段不能只检查 `rise ~= slope*duty + intercept` 的线性回归拟合，还应检查残差是否随 duty 系统性变化。如果高 duty 阶段的稳态温升明显低于线性预测，说明散热或功率链已经非线性。

### 5.4 阶段之间的冷却操作破坏了独立阶跃假设

当前固件不是每个 duty 都从相同温度和相同内部热状态开始。若使用阶跃辨识，推荐两种实验之一：

1. 每个 duty 独立从充分冷却的相同初始状态开始，重复多次；
2. 不切段，直接使用连续输入轨迹做状态空间/预测误差辨识，将冷却 duty 也作为输入的一部分。

不建议把一条包含多个未充分冷却阶段的曲线，直接当成多个相互独立的标准阶跃响应。

### 5.5 温度日志本身可能扰动被测系统

当前普通模式通过 `LOG_INF` 输出温度和 duty，自动辨识阶段也通过 `LOG_INF` 输出阶段数据。日志可能产生：

- CPU/线程时间开销；
- UART 缓冲和阻塞；
- 线程调度抖动；
- 采样与打印不在同一个时刻发生。

建议在正式辨识时采用二进制环形缓冲或低频批量导出，至少将“控制更新时间”和“日志输出时间”分开，不要把串口到达时间当作样本发生时间。

## 6. 仿真脚本的具体问题

### 6.1 植物离散化是近似 Euler，延迟和采样索引没有统一定义

两个脚本都使用：

```text
T[k+1] = T[k] + (T_eq[k] - T[k]) * dt / tau
```

当 `dt << tau` 时，Euler 近似通常可以接受，但应明确 `u[k]` 到底影响 `T[k]` 还是 `T[k+1]`。当前代码在不同脚本中采用了不同的 buffer 长度和更新顺序：

- `imu_ctrl_log.py`：`delay_n=max(1, round(delay/dt))`；
- `tune_imu_pi.py`：`delay_samples=max(0, round(delay/dt))`，buffer 长度是 `delay_samples+1`。

这两个实现对同一个 `delay=0` 或 `delay=1.793` 不一定产生同样的有效离散延迟。拟合脚本得到的参数送入 PI 脚本后，闭环响应可能因此发生偏差。

建议统一为一个显式的离散植物函数，并用解析离散化：

```text
a = exp(-dt/tau)
T[k+1] = a*T[k] + (1-a)*(Tambient + K*u[k-N])
```

其中 `N` 的定义、初始 buffer、输入更新时刻必须写进测试用例。

### 6.2 闭环理论公式忽略了纯延迟

位置：`scripts/imu/imu_ctrl_log.py:312-315`。

脚本打印的二阶近似：

```text
T(s) = C(s)G(s) / (1 + C(s)G(s))
```

实际把 `exp(-delay*s)` 从公式中省略了。这可以作为“忽略延迟的低阶近似”，但不能拿来计算真实闭环阻尼比并据此判断稳定性。纯延迟会额外引入相位滞后，可能显著降低相位裕度。

此外，输出饱和、积分清零、防卡角和最小 duty 限幅都会使系统成为非线性系统，不能只靠这个线性二阶公式描述。

### 6.3 稳定时间指标没有要求“连续保持”

位置：`scripts/imu/tune_imu_pi.py:167-174`。

实现方式取最后一次超出误差带的位置，再把稳定时间设为它之后的时刻。这在离线数组上通常能表达“从某时刻以后没有再出带”，但报告指标没有明确连续保持窗口长度，也没有分别处理：

- 温度进入误差带后短暂穿出；
- 温度刚好在仿真结束前进入误差带；
- 采样噪声造成的单点穿出；
- 最后 10 秒稳态误差和稳定带定义不一致。

建议定义固定验收语义：进入 `±0.2 C` 后连续 `N` 秒保持，且在窗口内最大绝对误差、RMSE、斜率分别满足阈值。

### 6.4 评分权重没有物理归一化

位置：`scripts/imu/tune_imu_pi.py:207-216`。

当前 score 将摄氏度、秒和比例直接乘不同权重相加。除非明确这些权重代表业务优先级，否则 score 的排名只是人为尺度的结果。尤其 `steady_error` 权重 100、`deadline_penalty` 1000，可能让程序偏好某些响应，但未必符合“不过冲、不过热、寿命和稳定时间”的实际优先级。

建议先输出所有原始指标，再使用硬约束筛选：

```text
安全温度不超过上限
最大 duty 不超过限制
稳态误差 <= 规定值
连续保持时间 <= 规定值
```

硬约束通过后，再用归一化指标做排序。

## 7. 什么才算辨识正确

“正确”不能只定义为拟合曲线贴合当前数据，而应至少包含以下五层。

### 7.1 数据层正确

- 时间单调递增，异常间隔被报告而不是静默吞掉；
- 温度和 duty 具有同一采样时刻定义；
- 阶段开始/结束边界可追溯；
- 没有严重日志丢帧或串口阻塞；
- 输入 duty 有足够激励。

### 7.2 训练拟合层正确

- 模型在拟合数据上的 RMSE、MAE 满足目标；
- 残差没有明显趋势、周期性或随温度/duty 变化的结构；
- 参数在不同初值下收敛到相近区域；
- 估计出的参数有合理范围和置信区间；
- 更复杂模型没有只靠增加自由度过拟合。

### 7.3 独立数据层正确

用另一轮实验、不同 duty 序列、不同起始温度验证。至少报告：

- 预测 RMSE/MAE；
- 最大误差；
- 首次到达目标温度的时间误差；
- 稳态温度误差；
- 预测延迟误差；
- 不同温度段的误差。

如果模型只在原始日志上好、在新日志上明显变差，它就是过拟合或实验条件依赖模型。

### 7.4 闭环层正确

把同一组 PI 参数、目标温度、初始温度和 duty 限制输入仿真，再与真机闭环对比。至少要求：

- 首次到达时间误差在规定范围内；
- 最大超调误差在规定范围内；
- 稳定时间误差在规定范围内；
- 稳态 duty 和温度误差可接受；
- 模型没有漏报饱和、振荡或慢性漂移。

### 7.5 鲁棒性层正确

在不同环境温度、不同初始温度、不同供电/安装条件下重复验证。若参数随条件变化明显，应承认模型是局部模型，并考虑增益调度或温度相关模型，而不是强行寻找一组“万能参数”。

## 8. 推荐的验证实验

### 8.1 实验 A：采集链路自检

目的：先确认数据不是坏的。

记录字段建议为：

```text
sample_index
monotonic_time_us
temperature_c
duty_command
actual_pwm_pulse_or_duty
control_dt_us
stage_id
stage_event
ambient_or_board_reference_temperature
```

验收：

- `time` 严格递增；
- `dt` 中位数、P95、最大值均输出；
- 日志丢帧可计数；
- duty 的更新时间和温度采样更新时间有清楚定义；
- 日志输出不会阻塞控制循环。

### 8.2 实验 B：单次开环阶跃

从充分冷却且热状态可重复的条件开始，固定 duty，例如：

```text
0.001 -> 0.20 -> 0.30 -> 0.40
```

每个阶段维持足够长时间，直到满足“温度斜率 + 方差 + 持续时间”的稳态标准。每个阶跃至少重复 3 次。

注意：这不是让 PI 控制器运行到目标温度，而是直接把 duty 作为输入激励。若硬件安全要求不能长时间开环，应使用限温保护和最大持续时间。

### 8.3 实验 C：多级输入或 PRBS

单阶跃只能较好估计总增益和主时间常数，对 delay 和多个时间常数的分离能力有限。建议在安全范围内使用：

- 多级阶跃；
- 小幅 PRBS；
- 随机但限幅的 duty 序列。

输入必须覆盖实际闭环工作区间，且不要长期卡在 duty 最小/最大限幅。

### 8.4 实验 D：独立验证轨迹

不要把同一条轨迹同时用于拟合和验证。推荐划分：

```text
实验 1、2：拟合
实验 3：独立开环验证
实验 4：闭环 PI 验证
实验 5：换起始温度/环境温度的鲁棒性验证
```

### 8.5 实验 E：闭环回放

记录真机闭环的完整状态：温度、目标、控制器原始输出、限幅后 duty、积分状态、实际 dt。将真机 duty 送入候选植物模型，先验证“输入回放”；再将候选植物和固件等价 PI 串起来，验证“闭环预测”。

这两步要分开，否则无法判断误差来自植物模型还是控制器仿真不一致。

## 9. 分阶段执行计划

### 阶段 0：基线冻结，预计 0.5 天

目标：保留当前结果，避免修脚本后无法解释历史参数。

工作项：

- 保存当前脚本版本和现有 `K=41.797`、`tau=10.408`、`delay=1.793`；
- 保存至少一份原始串口日志；
- 记录当前固件 PI 配置和板级实验条件；
- 不把当前参数称为“已验证模型”，暂称“历史参考模型”。

输出：一份不可变原始日志、一次当前脚本输出和一张当前拟合图。

### 阶段 1：修数据协议，预计 1 天

目标：让每个样本的时间和控制语义可追溯。

建议最小改动：

- 给每条样本增加单调时间或 cycle 时间，而不是只依赖日志显示时间戳；
- 记录 `sample_index`、`control_dt`、`stage_id`、事件行；
- 解析时检查时间倒退、重复、异常大间隔；
- 在 Python 中输出有效样本数、丢帧数、dt 分位数；
- 修复 `AMBIENT_TEMPERATURE` 配置未使用的问题；
- 区分“原始样本”与“用于显示的平均样本”。

完成标准：任何一条样本都能回答“什么时候采到、当时输入是什么、属于哪个阶段”。

### 阶段 2：修固件辨识状态机，预计 1 天

目标：阶段结束真正代表热状态足够稳定。

建议判据：

- 使用固定长度窗口，例如 5 到 10 秒；
- 对窗口内温度做线性回归，限制 `abs(slope)`；
- 同时限制标准差和均值变化；
- 记录阶段开始温度、阶段结束温度、稳定窗口长度、斜率；
- 每阶段使用固定最大时长和安全温度；
- 明确下一阶段是否要求回到统一初始温度，或者改用连续轨迹辨识。

完成标准：同一 duty 重复实验时，阶段结束温度和结束斜率具有可重复性。

### 阶段 3：先建立开环一阶基线，预计 1 到 2 天

目标：不引入复杂模型，先确认最简单模型能否预测。

工作项：

- 每个 duty 独立重复至少 3 次；
- 使用真实时间的离散模型；
- 分别拟合无延迟一阶和带延迟一阶；
- 对 `K/tau/delay` 做边界和多初值优化；
- 输出训练误差、验证误差、残差图和参数稳定性。

完成标准：独立开环轨迹的误差达到预先设定阈值，且残差没有明显结构。

### 阶段 4：比较一阶和二阶模型，预计 1 天

只有阶段 3 证明一阶模型不足时才做。

比较内容：

- 一阶无延迟；
- 一阶加延迟；
- 二阶串联加延迟；
- 必要时加入温度相关散热项。

模型选择不能只看训练 MSE，应看独立验证误差、参数可重复性、闭环关键指标和复杂度。

### 阶段 5：统一植物与 PI 仿真，预计 1 天

目标：让 Python 仿真和固件语义一致。

工作项：

- 抽出唯一的离散植物实现；
- 所有积分使用传入 `dt`；
- 明确定义输入、输出、delay 的样本索引；
- 复现 PID 的积分限幅、输出限幅、误差反向清零和最小 duty；
- 真正实现 `kp/ki` 网格扫描；
- 输出 Pareto 表，而不只输出单一 score 最优值；
- 添加“模型回放”和“闭环仿真”两种模式。

完成标准：用同一组离散输入，两个脚本的植物输出逐点一致；固定 PI 的仿真可以用单元测试锁定。

### 阶段 6：真机闭环交叉验证，预计 1 到 2 天

目标：验证模型不是只对开环数据有效。

工作项：

- 使用未参与辨识的初始温度和环境条件；
- 使用历史 PI 和候选 PI 各做至少 3 次；
- 对比首次到达、超调、稳定时间、稳态温度、稳态 duty；
- 记录安全上限和饱和比例；
- 若偏差随温度或 duty 系统性变化，建立局部模型或增益调度。

完成标准：闭环关键指标均在目标误差范围内，且没有依赖某一次偶然实验。

## 12. 当前数据输出协议与运行方式

### 12.1 固件输出

普通闭环模式每约 100 个 IMU 样本输出一条：

```text
imu_ctrl,seq=1200,t_ms=123456,temp=39.8421,duty=0.18400,dt=0.001012
```

自动辨识模式只输出 `heater_id` 数据，阶段加热样本约 10 Hz：

```text
heater_id,state=heating,stage=0,seq=1200,t_ms=123456,temp=36.8421,duty=0.35000,dt=0.001012,stable=143
```

阶段事件还包括：

```text
heater_id,stage_start,stage=0,seq=1000,t_ms=123000,target=0.35000,temp=36.0000,dt=0.001000
heater_id,stage_end,stage=0,seq=8000,t_ms=131000,target=0.35000,temp=39.2000,dt=0.001010,stable=1,span=0.04000
heater_id,sweep_finished,stage=2,seq=24000,t_ms=147000,temp=45.1000,safety=0
```

字段含义：

| 字段 | 含义 | 用途 |
|---|---|---|
| `seq` | 成功读取样本序号 | 检查是否丢样本 |
| `t_ms` | MCU 单调时间，单位 ms | 作为样本真实时间轴 |
| `temp` | 温度，单位 C | 模型输出 |
| `duty` | 实际施加的 PWM 占空比 | 模型输入 |
| `dt` | 两次成功采样之间的实际时间，单位 s | 检查采样抖动和离散化 |
| `stage` | 自动辨识阶段编号 | 分段和事件对齐 |
| `stable` | 稳定窗口内样本计数 | 判断阶段状态 |

注意：`kIdStartTemp=36 C` 在当前实现中是冷却起始温度上限，不再要求温度必须落在 `35.9~36.1 C`。实际每个阶段的起始温度以 `stage_start,temp` 为准；这是因为不同板卡、环境温度和安装条件下，最小 duty 的冷却平衡温度可能不同。

### 12.2 自动辨识采集

自动辨识入口由 `project/thread/Kconfig` 中的 `TRD_IMU_IDENTIFICATION` 控制。当前工作区实际配置为 `default y`，所以默认会进入辨识模式；普通闭环运行必须显式关闭。实验配置如下：

```conf
CONFIG_TRD_IMU=y
CONFIG_TRD_IMU_IDENTIFICATION=y
```

普通闭环运行不要打开该选项，否则 heater 会进入 duty 阶梯辨识，不再执行 40 C PI 控温。

先把 `scripts/imu/imu_temp_identify.py` 的串口配置为实际端口，然后运行：

```powershell
python scripts/imu/imu_temp_identify.py --port COM21 --baud 921600 --output imu_heater_id.csv --no-plot
```

如果使用已有原始串口日志：

```powershell
python scripts/imu/imu_temp_identify.py imu_heater_raw.log --output imu_heater_id.csv --no-plot
```

`imu_heater_id.csv` 会保存每个阶段的相对时间、温度、duty 和 dt。拟合前先看每个阶段的 `T0`、`Tss`、斜率、`tau` 和 RMSE，不要只看共享 `tau/L`。

### 12.3 普通闭环采集

`scripts/imu/imu_ctrl_log.py` 当前使用文件顶部配置：

```python
SERIAL_PORT = "COM21"
SERIAL_BAUD = 921600
LOG_FILE = None
OUTPUT_CSV = "imu_ctrl_capture.csv"
AMBIENT_TEMPERATURE = None
```

其中 `LOG_FILE = None` 表示从串口读取；设置为日志文件路径则离线解析。运行后会先保存 `imu_ctrl_capture.csv`，再进行拟合。若已知实验前的环境温度，应明确填写 `AMBIENT_TEMPERATURE`，不要让脚本用第一条可能仍在变化的温度代替环境温度。

### 12.4 数据使用顺序

```text
原始串口输出
    -> imu_heater_id.csv
    -> 检查 seq/t_ms/dt/stage
    -> 分阶段曲线和共享形状拟合
    -> 独立实验验证
    -> 再将 K/tau/delay 放入 tune_imu_pi.py
```

当前不要把 `imu_ctrl_capture.csv` 的闭环数据和 `imu_heater_id.csv` 的开环辨识数据混在一起拟合。前者用于闭环输入回放和闭环验证，后者才是优先用于植物参数辨识的数据。

本轮代码改动对应的实际位置：

- `modules/imu/imu.hpp`：为 `Sample` 增加 `sequence/timestamp_ms`，增加 5 秒稳定窗口、斜率/温差阈值和日志周期状态。
- `modules/imu/imu.cpp`：为成功样本写入 MCU 单调时间和序号；普通模式输出 `imu_ctrl`；辨识模式输出带阶段信息的 `heater_id`；阶段稳定判定改为窗口斜率判定。
- `scripts/imu/imu_ctrl_log.py`：优先解析 `t_ms`，检查时间单调性，保存闭环 CSV，支持 CSV 回放，并实际使用环境温度配置。
- `scripts/imu/imu_temp_identify.py`：兼容新的 `stage/seq/t_ms` 事件格式，支持 `--output` 保存分阶段 CSV。
- `scripts/imu/tune_imu_pi.py`：积分使用传入的 `dt`，并实际执行 `kp/ki` 网格扫描。
