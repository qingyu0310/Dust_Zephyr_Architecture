# 三个嵌入式框架对比总结

## 对比对象

本文对比三个我当前代码演进阶段中具有代表性的项目：

1. `temp/basic_framework-master`
2. `temp/Dust_SentinelRobot_L_Game`
3. 当前仓库 `tflm`

它们不是简单的“谁更高级”，而是分别代表了不同阶段、不同目标下的架构选择。

---

## 一句话结论

- `basic_framework-master` 最像一套成熟的 STM32/F4 队伍级基础平台，规则和文档最完整。
- `Dust_SentinelRobot_L_Game` 最像围绕一台具体机器人快速打通的实战工程，业务闭环最强。
- 当前 `tflm` 最像把项目层从框架层里独立出来的新一代平台骨架，职责边界最清晰，项目迁移潜力最大。

如果把三者放到一条演进线上看，可以理解为：

`basic_framework -> Dust_SentinelRobot_L_Game -> tflm`

前者提供“规则化框架视角”，中间提供“整机落地经验”，后者则开始形成“自己的架构语言”。

---

## 一、basic_framework-master

### 定位

这是一个偏“队伍级通用底座”的框架，目标不是只服务某一台机器人，而是服务多个机器人、多个成员、多个赛季的复用和传承。

### 优点

#### 1. 规则最完整

它明确提出了 `bsp / module / app` 三层结构，并且不是只停留在目录命名，而是把设计思想、任务频率、数据流、初始化流程、单双板切换策略都写进了文档。

这意味着它最强的地方不只是代码，而是“制度”。

#### 2. 文档体系最成熟

它的 README、`.Doc`、各层说明文档、模块说明文档都很齐全，明显是面向“新人可上手、队伍可传承”去设计的。

这一点非常重要，因为很多嵌入式项目代码能跑，但没人能接。

#### 3. 应用层解耦意识很强

它在 app 层主动引入 `message_center` 这种 pub-sub 机制，强调 `robot_cmd / chassis / gimbal / shoot` 之间不要直接相互包含，而要通过消息交换。

对于 C 风格 STM32 工程来说，这已经是很明显的架构意识了。

#### 4. 多机器人、多板卡复用能力强

它通过 `robot_def.h` 做单双板、兵种、参数切换，这种方式虽然不算最先进，但很适合“一个队伍多台车共用一套底座”。

### 缺点

#### 1. 规则强，但仍偏手工约束

很多边界是靠文档、命名和开发纪律维持，而不是靠语言机制或更强的编译期约束维持。

例如：

- 大量宏配置
- 手工注册 topic
- 手工安排任务频率
- 手工维护参数和外设归属

这使它很强，但也容易随着规模增长出现“规则知道，但执行走样”的问题。

#### 2. 与 STM32 HAL / CubeMX 耦合较深

它本质上还是一套基于 STM32F4 + CubeMX + FreeRTOS 的平台，移植性主要体现在“同类单片机工程”内，而不是更高层面的平台无关。

#### 3. pub-sub 已经有了，但机制本身还比较轻

`message_center` 更像一个工程内消息中间件，而不是非常严格的数据总线机制。

优点是轻、直接、够用；缺点是当系统继续膨胀时，数据一致性、队列语义、并发保护可能会成为进一步演进的瓶颈。

### 适合的场景

- 队伍传承
- 多机器人共底座
- 给新人提供一套能直接上手的工程平台
- 以 STM32 裸机/HAL/FreeRTOS 为主的软件栈

---

## 二、Dust_SentinelRobot_L_Game

### 定位

这是一个明显偏“具体机器人落地工程”的项目。

它虽然也有框架意识，但核心目标不是打造一套长期通用 SDK，而是围绕一台哨兵机器人，把底盘、云台、装填、裁判系统、上下板通信等全部打通。

### 优点

#### 1. 业务闭环最强

这个项目最大的优点是“真能打”。

从启动链路到总控逻辑，再到底盘、云台、通信、裁判系统、超级电容，已经是完整的整机工程，而不是只把底层模块封装一下。

#### 2. Robot 总控非常直观

它通过 `Robot` 作为整机抽象，把：

- `McuComm`
- `Chassis`
- `Gimbal`
- `Reload`
- `Imu`
- `Supercap`
- `Referee`

这些核心对象都装在一起，再用 `Robot::Task()` 完成跨模块决策。

这种设计虽然不是最解耦，但在整机项目里非常直观、非常高效。

#### 3. 控制逻辑已经不是“玩具级”

像底盘部分已经有：

- 工作模式切换
- 坐标变换
- 斜坡规划
- 运动学逆解
- 功率控制

云台部分也有：

- 角度环
- 速度环
- 滤波
- 电机失联恢复

这说明它不只是“会调 PID”，而是已经在组织完整控制链了。

#### 4. 落地经验浓

它的代码里有很多工程味很重的东西：

- CAN 回调直接分发到具体对象
- 任务内临界区复制共享数据
- 外设和业务对象强绑定
- 失联保护和运行模式切换

这些都很像真实机器人项目，而不是实验性质工程。

### 缺点

#### 1. 总控强，但总控偏重

`Robot` 很像“大脑”，很多跨模块策略都堆在里面。

短期看这很好，因为逻辑集中、调试方便；长期看容易变成“超大总控文件”，后续拆分难度会上升。

#### 2. 边界更多靠作者脑内维持

它有结构，但没有像 `basic_framework` 那样把规则制度化得特别清楚。

所以它更像“优秀的具体项目”，而不是“可广泛继承的平台”。

#### 3. 复用性不如另两者

里面很多设计都围绕当前这台哨兵机器人的设备拓扑和任务关系展开，迁移到其他机器人时，需要作者自己做较多判断和裁剪。

### 适合的场景

- 某一台具体机器人快速迭代
- 团队中有较强主导者，能持续维持整体结构
- 追求整机联调效率和实战闭环，而不是先做平台化

## 补充：basic_framework 与 Dust 的直接横向对比

前面的定位能够说明两套架构分别擅长什么，但还不够说明它们在代码组织上的根本差异。把两套工程放在一起看，最重要的不是 `C` 和 `C++` 的区别，也不是目录名字是否相似，而是它们选择了不同的“系统组织中心”：

- `basic_framework-master` 以**层级、实例和消息契约**组织系统；
- `Dust_SentinelRobot_L_Game` 以**整机对象、设备拓扑和总控任务**组织系统。

这也是为什么两套架构都能完成机器人控制，但阅读体验、扩展方式和迁移成本完全不同。

### 1. 目标不同：一个先做平台，一个先做整机

`basic_framework-master` 的目标从文档里就写得很清楚：它希望成为队伍长期复用的 SDK，面向多个机器人、多个成员和多个赛季。它预设了“底层能力应该沉淀，新的机器人主要重写应用层”这一使用方式。

Dust 的目标则更接近“把这一台哨兵真正做出来”。它把上下板通信、裁判系统、底盘、云台、拨弹盘、IMU、超级电容和自动瞄准数据都放进同一个整机工程里，优先保证设备拓扑和业务闭环能跑通。

因此：

| 对比项 | `basic_framework-master` | `Dust_SentinelRobot_L_Game` |
| --- | --- | --- |
| 第一目标 | 队伍级复用与传承 | 当前机器人落地与联调 |
| 组织单位 | `bsp / module / app` | `Robot` 及其成员对象 |
| 主要变化点 | 新机器人主要改 `app` 和参数 | 新机器人通常需要重新组织整机对象和业务链 |
| 评价标准 | 是否容易复用、交接、扩展 | 是否能快速、完整、稳定地完成整机控制 |

这里没有绝对的优劣。前者优化的是“下一个项目”，后者优化的是“当前项目”。

### 2. 分层不同：HNU 的目录就是规则，Dust 的目录更像工程分区

在 HNU 架构中，`bsp / module / application` 不只是三个文件夹，而是明确的依赖方向：

```text
application
    ↓
module
    ↓
bsp
    ↓
STM32 HAL / CubeMX 外设
```

其中还有两种不同的组织关系：

1. `app` 包含并使用 `module`，例如 `chassis` 包含四个电机实例；
2. 顶层 `app` 之间不直接包含彼此，而是通过 `message_center` 进行发布和订阅。

所以它同时使用了“结构层级模式”和“应用并列的 pub-sub 模式”。`chassis` 可以包含电机模块，电机模块可以包含 CAN 实例，但 `chassis` 不应该直接包含 `gimbal` 或 `robot_cmd`。

Dust 虽然也有 `Bsp`、`Communication`、`Device`、`Modules`、`Algorithm`、`Interaction` 这些目录，但它们更多是工程功能分区，未完全形成一条由编译关系强制保证的依赖规则。真正的系统组织关系要看 `Interaction/Robot.h`、`Interaction/Robot.cpp` 和 `Interaction/Init.cpp`：

```text
Robot
 ├─ McuComm
 ├─ Chassis
 ├─ Gimbal
 ├─ Reload
 ├─ Imu
 ├─ Supercap
 └─ Referee
```

也就是说，Dust 的核心边界不是“应用彼此隔离”，而是“整机由哪些对象组成、对象之间如何协同”。目录分层有帮助，但最后仍然需要维护者理解整台机器人。

### 3. 初始化模型不同：HNU 集中编排，Dust 分散到对象

HNU 的启动链比较集中：

```text
main
  → RobotInit()
      → BSPInit()
      → RobotCMDInit()
      → GimbalInit()
      → ShootInit()
      → ChassisInit()
      → OSTaskInit()
  → 启动 FreeRTOS
```

`RobotInit()` 负责建立整机初始状态，`OSTaskInit()` 负责创建基础任务。应用初始化和任务创建的位置相对固定，阅读者可以从 `application/robot.c` 和 `application/robot_task.h` 进入整个系统。

Dust 的启动链来自 CubeMX 生成的 FreeRTOS 工程：

```text
main()
  → HAL / MX 外设初始化
  → osKernelInitialize()
  → MX_FREERTOS_Init()
  → StartDefaultTask()
      → Init()
          → can_init()
          → uart_init()
          → robot_.Init()
              → mcu_comm_.Init()
              → gimbal_.Init()
              → chassis_.Init()
              → reload_.Init()
              → supercap_.Init()
              → osThreadNew(Robot::TaskEntry, ...)
```

Dust 的一个鲜明特点是：每个对象在自己的 `Init()` 中创建自己的任务。例如 `Chassis::Init()` 创建 `chassis_task`，`Gimbal::Init()` 创建 `gimbal_task`，`Reload::Init()` 创建 `reload_task`，`Imu::Init()` 创建 `ImuTask`。这样做很贴近 C++ 对象模型，也便于把对象作为完整功能单元搬动；代价是系统任务不再只看一个统一的任务清单，任务创建点会分散在多个类中。

可以把两者概括为：

- HNU：**先建立整机任务骨架，再让应用运行**；
- Dust：**初始化每个功能对象时，同时让对象拥有自己的运行任务**。

### 4. 应用通信不同：消息中心解耦 vs 总控直接协调

这是两套架构差异最大的地方。

#### HNU：应用之间通过话题通信

`robot_cmd` 发布：

- `chassis_cmd`
- `gimbal_cmd`
- `shoot_cmd`

同时订阅：

- `chassis_feed`
- `gimbal_feed`
- `shoot_feed`

`chassis`、`gimbal`、`shoot` 各自只需要知道话题和消息结构，不需要包含 `robot_cmd` 的头文件，也不需要直接持有另一个应用的对象。这种关系可以画成：

```text
robot_cmd ──发布控制命令──> message_center ──> chassis / gimbal / shoot
robot_cmd <─订阅反馈数据── message_center <── chassis / gimbal / shoot
```

它的优点是应用可以并行开发、替换和裁剪；缺点是消息类型、话题名称、注册顺序和队列语义需要额外维护。当前实现的 `message_center` 使用字符串话题、订阅者链表和固定长度小队列，轻量直接，但并不是一个完整的数据总线。

#### Dust：中断按 ID 分发，总控直接读写对象

Dust 的 CAN 接收入口在 `Interaction/Init.cpp`。收到 CAN 帧后，回调根据 `StdId` 直接调用具体对象：

```text
CAN1
  0x201 → robot_.chassis_.motor_chassis_1_
  0x202 → robot_.chassis_.motor_chassis_2_
  0x203 → robot_.chassis_.motor_chassis_3_
  0x204 → robot_.chassis_.motor_chassis_4_
  0x205 → robot_.reload_.motor_reload_1_

CAN2
  0x01  → robot_.mcu_comm_
  0x07  → robot_.gimbal_.motor_yaw_
  0x30  → robot_.supercap_
```

`Robot::Task()` 再直接读取 `mcu_comm_` 的数据，调用 `gimbal_`、`chassis_` 和 `reload_` 的接口设置目标。跨模块决策集中在一个地方，例如：

- 根据上下板通信状态决定云台当前 IMU 角；
- 根据遥控器开关和自瞄模式计算云台目标角；
- 根据裁判系统状态切换底盘自旋或正常模式；
- 根据开火状态设置拨弹盘目标速度。

这种方式不需要经过话题查找、消息复制和订阅关系，数据路径短、调试直观，非常适合整机联调；但 `Init.cpp` 和 `Robot.cpp` 需要知道大量设备 ID、成员对象和业务规则，模块之间的耦合也因此集中到了总控位置。

### 5. 硬件抽象不同：实例注册 vs 具体对象封装

HNU 使用的是 C 风格实例模型。典型调用形态是：

```c
DJIMotorInstance *motor = DJIMotorInit(&config);
DJIMotorSetRef(motor, target);
```

模块通过 `XXXInstance` 保存状态，BSP 通过 `XXXInstance` 和回调指针找到对应的上层对象。这种方式的优势是接口形态统一，多个电机、多个 IMU、多个 CAN 实例都可以按照同样的注册模式创建。模块在没有被 app 实例化时，也可以保持未使用状态。

Dust 使用 C++ 类把设备封装成对象，例如：

```cpp
MotorDjiC620 motor_chassis_1_;
MotorDjiC620 motor_chassis_2_;
MotorDmNormal motor_yaw_;
```

对象直接保存自己的状态、PID、CAN 句柄和控制方法，CAN 回调也可以直接调用对象成员函数。这个模型比 C 风格实例更自然，尤其适合表达“这个电机属于这个底盘对象”这种所有关系。

但 Dust 的设备拓扑通常直接写在业务对象的初始化中，例如：

```cpp
motor_chassis_1_.Init(&hcan1, 0x201, ...);
motor_yaw_.Init(&hcan2, 0x07, 0x06);
```

因此它的对象封装更强，配置外置能力却弱一些。更换机器人时，不只是替换一个配置结构体，还要重新检查 `Robot` 成员、CAN ID、回调分发和业务关系。

### 6. 业务逻辑位置不同：平行应用 vs 整机总控

HNU 把业务拆在多个 app 中：

```text
robot_cmd
 ├─ 解析遥控器 / 视觉 / 上位机输入
 ├─ 发布底盘、云台、发射命令
 └─ 订阅各应用反馈

chassis
 ├─ 订阅底盘命令
 ├─ 运动学解算
 └─ 控制底盘电机

gimbal
 ├─ 订阅云台命令
 └─ 控制云台电机与 IMU

shoot
 ├─ 订阅发射命令
 └─ 控制摩擦轮、拨盘和弹舱
```

这会让每个 app 的职责更窄，也方便多人分别维护。但如果某个策略横跨多个 app，就必须设计新的消息结构或反馈路径。

Dust 则把横跨多个部件的策略直接写在 `Robot::Task()` 中。它不需要为“云台状态影响底盘模式”“裁判系统影响拨弹盘”额外设计一套消息流，直接调用即可。这正是它在比赛项目中高效的原因：

```text
上下板通信 / 裁判系统 / 自瞄数据
                  ↓
             Robot::Task()
          ↙         ↓        ↘
       Gimbal     Chassis    Reload
```

代价是总控会逐渐承担越来越多的策略。如果后续继续增加自动导航、能量机关、视觉状态机、更多故障模式，`Robot::Task()` 可能成为最需要拆分的地方。

### 7. 多板支持不同：配置切换 vs 工程内显式通信

HNU 把单双板视为框架能力，在 `application/robot_def.h` 中通过 `ONE_BOARD`、`CHASSIS_BOARD`、`GIMBAL_BOARD` 等宏切换。单板时，应用之间走 `message_center`；双板时，根据条件编译切换到 `CANComm`。同一套 app 代码通过编译条件适配不同板卡角色。

Dust 的上下板通信是当前整机的一部分。`McuComm` 直接拥有 CAN 通信状态、收发数据结构和失联判断，`Robot::Task()` 直接消费这些数据。它不是在设计一个通用的“板卡角色模型”，而是在实现“这台哨兵的主控板和底盘板如何通信”。

所以：

- HNU 把多板看成**可配置的平台维度**；
- Dust 把多板看成**当前机器人业务的一条通信链路**。

### 8. 构建和移植不同：HNU 偏代码复用，Dust 偏工程复制

HNU 明确鼓励从框架 fork 出新的机器人工程，底层 `bsp/module` 更新后再通过 commit 合入。它的复用边界大致是：

```text
稳定复用：bsp + module
按机器人变化：application + robot_def.h
```

Dust 的 `CMakeLists.txt` 使用 `GLOB_RECURSE` 把 `Algorithm`、`Bsp`、`Communication`、`Device`、`Interaction`、`Modules` 等整个工程编进来。它更像一个完整的产品工程，迁移时通常是复制工程后改造 `Interaction`、设备配置和业务逻辑，而不是只替换某一层。

这不是 Dust 做得不够好，而是它的边界服务于另一个目标：项目文件可以快速加入编译链，新的设备和业务可以直接挂到现有工程中，整机工程不需要先完成平台化拆分。

### 9. 文档和传承不同：制度化说明 vs 代码经验

`basic_framework-master` 的架构能力有很大一部分来自文档：

- 根目录 `README.md` 说明总体定位；
- `.Doc/架构介绍与开发指南.md` 说明分层、初始化和依赖方向；
- `application/APP层应用编写指引.md` 说明 app 之间如何通信；
- `modules/message_center/message_center.md` 说明消息中心的注册、发布和获取；
- 各个 BSP、module 和 app 目录还有对应的使用文档。

Dust 的根目录 `Readme.md` 主要记录仓库用途、提交格式、板型命名和版本约定，架构知识主要沉淀在代码本身，尤其是 `Interaction/Robot.*`、`Interaction/Init.*` 和各个 `app_*.cpp` 的实现中。

因此，HNU 更适合新人按照阅读顺序学习和接手；Dust 更适合熟悉整台机器人、能沿着代码和设备拓扑直接追问题的维护者。

### 10. 最终判断

如果只比较“平台化能力”：

```text
tflm > basic_framework-master > Dust_SentinelRobot_L_Game
```

如果只比较“当前整机业务闭环”：

```text
Dust_SentinelRobot_L_Game > basic_framework-master
```

如果比较“跨项目迁移的直接成本”：

- HNU 更像“保留框架，重写 app”；
- Dust 更像“复制整机工程，再按新机器人重组”。

如果比较“复杂业务的短期开发效率”：

- Dust 的 `Robot` 直接协调方式更快；
- HNU 的消息中心和应用边界需要先设计消息契约，但长期协作更稳定。

最准确的总结不是“Dust 不如 basic_framework”，而是：

> `basic_framework-master` 把经验提炼成了平台规则；`Dust_SentinelRobot_L_Game` 把规则和经验压缩成了一台机器人能够运行的整机系统。

当前 `tflm` 真正值得继承的方向，正好是把两者的优势拆开吸收：

- 保留 HNU 的边界、文档、消息和复用意识；
- 保留 Dust 的整机控制链、设备对象和比赛实战经验；
- 再用 `project` 层承接新项目和移植项目，让新增机器人尽量只改项目装配，而不是修改框架层。

### 11. 按架构能力重新打分

上一版如果把 `basic_framework-master` 排在 `tflm` 前面，是评分标准错了。那种排法把“F4 工程内部已经成熟”和“框架是否更适合作为跨项目平台”混在了一起，也把文档、命名和代码规范的数量误当成了维护传承能力。

这里改用五个真正影响架构长期价值的维度，每项 20 分：

1. **运行时与平台基础**：是否站在更强的 RTOS、构建系统和硬件抽象之上；
2. **职责边界**：层之间的责任和依赖方向是否稳定；
3. **项目迁移能力**：新建项目或移植项目需要改动多少框架层代码；
4. **边界驱动的维护传承**：换人后能否只根据职责继续开发，而不是依赖原作者脑内结构；
5. **真实业务闭环**：能否直接承载完整机器人控制和比赛业务。

| 架构 | 平台基础 | 职责边界 | 项目迁移 | 边界维护传承 | 业务闭环 | 框架架构分 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 当前 `tflm` | 19 | 20 | 20 | 19 | 15 | **93** |
| `basic_framework-master` | 15 | 18 | 16 | 17 | 18 | **84** |
| `Dust_SentinelRobot_L_Game` | 13 | 13 | 10 | 13 | 20 | **69** |

#### 为什么 `tflm` 是 93 分

- 底层运行时已经站在 Zephyr 之上，不再局限于 STM32F4、HAL、CubeMX 和 FreeRTOS 的单一工程范式；
- `drivers / modules / algorithm / topic / project` 的职责边界比两套 F4 工程更明确；
- 在框架能力覆盖需求的前提下，新建项目或移植项目主要修改 `project/`；
- `project/apps`、`project/boards`、`project/thread` 负责装配，框架层不反向依赖项目层；
- `trd_test` 还能作为独立 Demo 和实验入口，验证完成后再把稳定能力沉淀回框架层。

它没有拿满分，主要是当前仍处于实现收敛期，部分历史代码、模块职责和线程装配点还在逐步统一；这是“现在还没完全收拢”的问题，不是架构方向的问题。

#### 为什么 `basic_framework-master` 是 84 分

- `bsp / module / app` 边界明确，应用之间还有 `message_center` 解耦；
- 新机器人主要改 `app` 和 `robot_def.h`，已经具备明显的平台复用意识；
- 但底层仍然深度依赖 STM32F4、HAL、CubeMX 和 FreeRTOS；
- 项目差异主要依靠宏、目录约定和人工维护，缺少 `project/` 这种把项目单元整体隔离出来的装配层；
- 它的文档很多是优点，但真正拉开分数的不是文档数量，而是它已经形成了较稳定的职责边界。

#### 为什么 `Dust_SentinelRobot_L_Game` 是 69 分

- 作为一台比赛机器人，它的业务闭环是三者里最完整的；
- 但它的整机对象、CAN ID、任务和跨模块决策集中在具体工程中；
- 新机器人迁移时通常需要重新组织 `Robot`、设备拓扑和业务链；
- 它有目录分区，但没有把职责边界提升为框架层和项目层之间的稳定契约。

#### 如果只评价“单台比赛整机”

框架架构分不能否定 Dust 的实战价值。换成“当前一台比赛机器人能否快速做完整、联调和交付”的评分，排序应该是：

| 架构 | 单台比赛整机分 | 说明 |
| --- | ---: | --- |
| `Dust_SentinelRobot_L_Game` | **95** | 总控直接，设备链完整，实战闭环最强 |
| `basic_framework-master` | **88** | 有成熟 app 和模块底座，但组装约束更多 |
| 当前 `tflm` | **80** | 框架上限最高，但当前具体整机业务仍在装配和收敛 |

所以正确的双排名是：

```text
框架架构能力：tflm > basic_framework-master > Dust_SentinelRobot_L_Game
单台比赛整机能力：Dust_SentinelRobot_L_Game > basic_framework-master > tflm
```

维护传承也应该这样理解：

> 不是文件规范越多、文档越厚，维护传承就越强；只要职责边界稳定，项目差异有明确归属，后来者就能沿着边界继续写。

`tflm` 当前最重要的价值，正是把这个条件进一步明确成了：

```text
框架层：drivers / modules / algorithm / topic
项目层：project/apps / project/boards / project/thread
```

## 三、当前 tflm

### 定位

当前 `tflm` 更像是对前两个阶段经验的“再抽象”，但它现在已经不只是一个“想法上的下一代框架”，而是一套**以 Zephyr 为运行时基础、以 Kconfig/CMake 为编译期裁剪手段、以 `drivers / modules / algorithm / topic / project` 为主要职责边界**的实际工程。

它已经不满足于“把机器人做出来”，而是在主动思考：

- 哪些是项目层
- 哪些是驱动层
- 哪些是模块层
- 哪些是算法层
- 哪些是消息层
- 哪些是板级配置

这说明它的目标更偏向“可迁移框架”。

### 优点

#### 1. 分层最清晰

当前仓库把结构拆成了：

- `project/`
- `drivers`
- `algorithm`
- `modules`
- `topic`

这比前两个 STM32 工程更干净，也更接近真正意义上的框架设计。

#### 2. 项目层与框架层分离得更主动

`README` 已经明确提出：

- `project/` 是项目单元
- `apps / boards / thread` 是 `project/` 的三部分
- 其余部分是集成环境

这是一个非常重要的抽象提升。

#### 3. 当前启动链已经不是老式初始化流程

当前 live tree 里的真实入口非常直接：

```text
src/main.c
  -> System_Startup()
     -> RunStage(Bsp)
     -> RunStage(ThreadEarly)
     -> RunStage(Module)
     -> RunStage(ThreadMid)
     -> RunStage(ThreadLate)
```

也就是说，现在的启动组织不是老的：

```text
System_Bsp_Init -> System_Modules_Init -> System_Thread_Start
```

真正的启动编排在 `project/apps/Init_entry.cpp`，`src/main.c` 只负责调用 `System_Startup()`。

这里最关键的不是“函数名字变了”，而是当前启动已经是**阶段式遍历器**，而不是手写的一长串固定初始化调用。

#### 4. `REGISTER_INIT()` + `.user_init` 让启动项本地注册

每个线程或组件都可以在自己的文件里声明：

```cpp
REGISTER_INIT(thread_init,  Module,     Low, "test_init");
REGISTER_INIT(thread_start, ThreadLate, Low, "test_start");
```

这些 `InitEntry` 会被链接到 `.user_init` 段，边界由 `cmd/linker/tflm_init.ld` 提供，运行时由 `Init_entry.cpp` 统一遍历。

这说明当前 `tflm` 的启动机制已经很明显地从“项目里手写一堆 init 顺序”，走到了“编译期注册 + 运行时按阶段调度”。

#### 5. 线程装配、依赖选择和编译裁剪已经连起来了

当前 `tflm` 已经把：

- `Kconfig`
- `CMake`
- feature toggle
- topic 编译条件
- thread 编译条件

结合起来了。

例如：

- `project/thread/Kconfig` 用 `select` 拉起模块、驱动和 topic 依赖；
- `project/thread/CMakeLists.txt` 决定线程源文件是否真正进入编译链；
- `project/thread/*/trd_xxx.cpp` 在本地实现 `thread_init()` / `thread_start()` 并用 `REGISTER_INIT()` 挂进启动阶段。

像当前仓库里：

- `trd_can_tx.cpp` 注册在 `Bsp` / `ThreadEarly`
- `trd_imu.cpp` 注册在 `Module` / `ThreadLate`
- `trd_test.cpp` 注册在 `Module` / `ThreadLate`

这已经不是“分文件夹”这么简单，而是真正在做按职责边界拆开的装配型架构。

#### 6. `project/` 和 `trd_test` 让迁移与实验都更自然

`project/` 被单独抽出来后，在框架层能力已经覆盖需求的前提下，新建项目或移植项目原则上主要只需要修改：

- `project/apps/`
- `project/boards/`
- `project/thread/`

同时 `trd_test` 也不是随便塞临时代码的角落，而是正式的 Demo / 设备验证 / 参数实验入口。当前它就已经在跑 DJI C610 的开环转矩注入 + RLS 辨识，这比“单独再搭一个实验工程”更符合这套架构的思路。

### 缺点

#### 1. 部分文档和注释还没完全同步到当前实现

这次对比里最典型的问题就是：旧描述很容易混进来。

例如：

- 有些旧文档还会写成 `projects/`
- 有些旧描述还会写老的初始化链
- 个别注释和当前真实启动顺序还没完全同步

所以看 `tflm` 时，当前代码应以：

- `src/main.c`
- `project/apps/Init_entry.hpp`
- `project/apps/Init_entry.cpp`
- `project/thread/*`

这些真实入口为准。

#### 2. 当前更像“正在长大的框架”

相比 `basic_framework-master` 那种已经长期服役的稳定底座，`tflm` 还处于不断演化阶段。

这不是坏事，只是说明它的主要价值在“未来潜力”而不在“历史稳定性”。

#### 3. 学习门槛反而更高

因为它的抽象层次更高，新人如果没有 STM32/RTOS/模块化思维基础，未必比 `basic_framework-master` 更容易直接理解。

### 适合的场景

- 想沉淀自己的框架方法论
- 想做跨项目、跨板卡、跨模块复用
- 想让架构边界更清晰
- 未来还希望继续接入更多算法、推理、板级适配能力

- 最强：`tflm`
- 次强：`basic_framework-master`

这里的“清晰度”是指概念边界是否干净，不是代码是否更复杂。

### 4. 从“边界驱动的维护传承”看

- 最强：`tflm`

- 职责边界是否稳定
- 项目差异是否有明确归属
- 后来者能不能沿着层边界继续开发

按这个标准，当前 `tflm` 更强，因为它把项目差异明确收敛到了 `project/`。

### 5. 从“文档完备度”看

- 最强：`basic_framework-master`
- 次强：`tflm`
- 第三：`Dust_SentinelRobot_L_Game`

如果只看文档体系、说明文档数量和新人阅读顺序，`basic_framework-master` 仍然最完整。

### 6. 从“未来扩展空间”看

- 最强：`tflm`
- 次强：`basic_framework-master`
- 第三：`Dust_SentinelRobot_L_Game`

因为 `tflm` 的结构更适合继续接入新的板级支持、算法模块、消息机制、构建裁剪和推理能力。

## 六、最后的结论

如果我要给三者下最简短的评价：

- `basic_framework-master`：最像一套成熟的 STM32/F4 队伍级 SDK。
- `Dust_SentinelRobot_L_Game`：最有实战气质，像一台机器人打出来的成熟工程。
- `tflm`：最像把项目装配从框架层剥离出来的下一代平台骨架。

如果只问“哪个最好”，这个问题其实不准确。

更准确的说法是：

- 想要一套成熟的 STM32/F4 队伍底座，`basic_framework-master` 很强。
- 想快速把一台机器人真正做出来，`Dust_SentinelRobot_L_Game` 很强。
- 想把项目差异尽量收敛到 `project/`，并继续往 Zephyr 平台化方向走，`tflm` 最强。

所以这三者合在一起，刚好能说明一条非常清楚的成长路径：

**从借鉴成熟框架，到独立完成整机落地，再到提炼出自己的框架体系。**
